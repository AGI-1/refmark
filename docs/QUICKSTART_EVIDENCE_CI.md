# Evidence CI Quickstart

This is the shortest product-shaped Refmark workflow:

```text
corpus -> stable refs -> retrieval eval -> comparison -> smells -> adaptation plan
       -> manifest diff when the corpus changes
```

The goal is not to replace your retriever. The goal is to give every retriever,
reranker, query rewriter, citation model, and reviewer the same concrete target:

```text
Did this run recover the correct source ref or range?
```

For the default local path, one command creates the standard artifacts:

```bash
python -m refmark.cli ci docs/ eval_questions.jsonl \
  --out-dir runs/refmark_ci \
  --min-hit-at-k 0.80 \
  --min-best-hit-at-k 0.80 \
  --fail-on-regression
```

The sections below show the same workflow step by step.

## 1. Map A Corpus

Create a shadow manifest. The source files do not need to be modified.

```bash
python -m refmark.cli map docs/ \
  -o .refmark/corpus.refmark.jsonl \
  --marked-dir .refmark/marked
```

The manifest maps stable refs such as `policy:P03` to source text, hashes,
paths, and ordinals. Treat it like a generated registry for the current corpus
revision.

If your production retriever already chunks documents, attach the relevant
stable refs to each chunk as metadata. Refmark does not require replacing the
chunker, embedding model, vector database, or search service.

## 2. Create Or Import Eval Rows

Use JSONL rows with `query` and `gold_refs`:

```jsonl
{"query":"How often should API tokens be rotated?","gold_refs":["security:P02"]}
{"query":"What happens to expedited shipping refunds?","gold_refs":["policy:P12-P13"]}
```

Gold refs can be curated by editors, generated by a question-generation agent,
or imported from another benchmark after mapping the evidence regions.

## 3. Build A Local Index

The built-in portable index is a no-infra baseline and demo artifact.

```bash
python -m refmark.cli build-index docs/ \
  -o runs/docs.index.json \
  --source local
```

You can still evaluate external retrievers through `eval-index
--retriever-results` or the Python `EvalSuite` API.

For an existing Python retrieval stack, the integration is just a callback:

```python
from refmark import CorpusMap, EvalSuite

corpus = CorpusMap.from_manifest(".refmark/corpus.refmark.jsonl")
suite = EvalSuite.from_jsonl("eval_questions.jsonl", corpus=corpus, attach_source_hashes=True)

def my_retriever(query: str):
    return [{"stable_ref": "security:P02", "score": 0.91}]

run = suite.evaluate(my_retriever, name="my_retriever", k=10)
print(run.metrics)
```

See `examples/library_integration_demo/run.py` for the same pattern with a
smell report, adaptation plan, and run artifact.

## 4. Run Evidence Eval

```bash
python -m refmark.cli eval-index runs/docs.index.json eval_questions.jsonl \
  --manifest .refmark/corpus.refmark.jsonl \
  --top-k 10 \
  --smell-report-output runs/smells.json \
  --adapt-plan-output runs/adaptation_plan.json \
  --min-hit-at-k 0.80 \
  --max-stale 0 \
  --fail-on-regression \
  -o runs/eval.json
```

The report includes:

- hit@1 / hit@k / MRR;
- gold coverage and region precision;
- validation of missing or ambiguous refs;
- stale examples when source hashes no longer match;
- data-smell diagnostics;
- provenance and comparable run fingerprints.

## 5. Compare Built-In Strategies

```bash
python -m refmark.cli compare-index runs/docs.index.json eval_questions.jsonl \
  --manifest .refmark/corpus.refmark.jsonl \
  --strategies flat,hierarchical,rerank \
  --top-k 10 \
  --min-best-hit-at-k 0.80 \
  --fail-on-regression \
  -o runs/compare_index.json
```

`refmark.compare_index_report.v1` keeps the corpus/eval fingerprints shared and
emits per-strategy metrics, smell summaries, run artifacts, and the current
best strategy by hit@k.

If runs already exist, compare the saved eval artifacts without rerunning
retrieval:

```bash
python -m refmark.cli compare-runs \
  runs/eval_bm25.json \
  runs/eval_embedding.json \
  runs/eval_hybrid.json \
  --baseline eval_bm25.json \
  --min-best-hit-at-k 0.80 \
  --fail-on-regression \
  -o runs/compare_runs.json
```

Use `compare-index` when you want Refmark to run built-in index strategies now.
Use `compare-runs` when the reports came from separate jobs, external
retrievers, hosted services, or previous CI artifacts.

## 6. Review Smells And Adapt

`runs/smells.json` is a review queue, not an automatic verdict. It can surface:

- stale labels;
- hard refs;
- repeated gold-vs-top confusions;
- query-style gaps;
- overcitation and undercitation;
- low-confidence jumps;
- query magnets.

`runs/adaptation_plan.json` converts those signals into review-required actions
such as shadow metadata/doc2query additions, stale-label refresh, range/context
tuning, confidence gating, and query-magnet role assignment.

Example smell row:

```json
{
  "type": "hard_ref",
  "ref": "security:P03",
  "severity": "medium",
  "evidence": {"miss_count": 4, "query_styles": ["concern", "direct"]}
}
```

Example adaptation action:

```json
{
  "adaptation_type": "shadow_metadata",
  "target_refs": ["security:P03"],
  "review_required": true,
  "rationale": "Gold ref is repeatedly missed; review aliases or doc2query metadata."
}
```

## 7. Check Corpus Changes

When the corpus changes, map it again:

```bash
python -m refmark.cli map docs/ \
  -o .refmark/corpus.rev-b.refmark.jsonl
```

Then compare manifests:

```bash
python -m refmark.cli manifest-diff \
  .refmark/corpus.rev-a.refmark.jsonl \
  .refmark/corpus.rev-b.refmark.jsonl \
  --examples eval_questions.rev-a.jsonl \
  --max-stale 0 \
  -o runs/manifest_diff.json
```

Use the heavier label validator when your eval rows carry source hashes:

```bash
python -m refmark.cli lifecycle-validate-labels \
  .refmark/corpus.rev-b.refmark.jsonl \
  eval_questions.rev-a.jsonl \
  --previous-manifest .refmark/corpus.rev-a.refmark.jsonl \
  --max-stale 0 \
  --max-removed-refs 0 \
  -o runs/lifecycle_report.json
```

## CI Shape

For GitHub Actions, keep this as a project-specific workflow skeleton rather
than a universal generated file:

```yaml
name: refmark-evidence-ci

on: [pull_request]

jobs:
  evidence:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install -e .
      - run: |
          mkdir -p runs
          python -m refmark.cli map docs/ -o runs/corpus.refmark.jsonl
          python -m refmark.cli build-index docs/ -o runs/docs.index.json --source local
          python -m refmark.cli eval-index runs/docs.index.json eval_questions.jsonl \
            --manifest runs/corpus.refmark.jsonl \
            --min-hit-at-k 0.80 \
            --max-stale 0 \
            --fail-on-regression \
            -o runs/eval.json
```

## What This Proves

This workflow proves that a corpus can be made addressable and regression-tested
by evidence refs. It does not prove that Refmark universally beats embeddings,
BM25, vector databases, or answer judges. Instead, it gives those systems a
shared measurement layer so teams can compare them, detect stale supervision,
and review concrete evidence failures.
